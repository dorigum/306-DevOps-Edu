# Limit 구문

## 1. LIMIT 구문 두 가지 형태

### (1) `LIMIT row_count OFFSET offset`

```sql
SELECT * FROM table
LIMIT 3 OFFSET 5;
```

- **OFFSET 5** → 처음 5행 건너뜀
- **LIMIT 3** → 그다음 3행 가져옴
- 즉, **6번째 행부터 3개** 반환

---

### (2) `LIMIT offset, row_count`

```sql
SELECT * FROM table
LIMIT 5, 3;
```

- **앞 숫자(5)** → offset (건너뛸 행 수)
- **뒤 숫자(3)** → row_count (가져올 행 수)
- 즉, **6번째 행부터 3개** 반환

## 2. 차이점

- **결과는 동일하다.**
    
    `LIMIT 5, 3` == `LIMIT 3 OFFSET 5`
    
- 단지 **표현 방식**이 다를 뿐이다.
    

---

## 3. 추천 사용 방식

- **가독성 측면**에서 `LIMIT n OFFSET m` 방식이 더 명확하다.
    
    (특히 페이징 처리할 때 직관적)
    
- 하지만 예전 MySQL 문법이나 간단한 쿼리에서는 `LIMIT offset, count`가 많이 쓰인다.
    

---

## 정리

- `LIMIT 5, 3` → offset 5, row_count 3 → **6번째 행부터 3개**
- `LIMIT 3 OFFSET 5` → 같은 의미 (표현 차이만 있음)
- 둘은 동작상 차이는 없고, **가독성**의 차이만 있다.

## 📌 1. MySQL 내부 동작

- MySQL 옵티마이저는 두 문법을 **동일한 쿼리 플랜**으로 변환해서 실행한다.
    
    → 즉, `LIMIT 5, 3`과 `LIMIT 3 OFFSET 5`는 실행 계획(Execution Plan)이 같음.
    
- 따라서 성능 차이는 전혀 없다.
    

---

## 📌 2. 문법적 차이

- `LIMIT offset, row_count` → **옛날 MySQL 문법** (전통적인 방식)
- `LIMIT row_count OFFSET offset` → **SQL 표준 방식** (PostgreSQL, Oracle, SQL Server 등에서도 유사 문법 존재)

👉 따라서 **이식성(다른 DBMS로 옮길 가능성)**을 고려하면 `LIMIT ... OFFSET ...` 쓰는 게 좋다.

---

## 📌 3. 가독성 차이

- `LIMIT 5, 3` → 앞 숫자가 offset인지 count인지 헷갈릴 수 있음
- `LIMIT 3 OFFSET 5` → `OFFSET` 키워드가 있어서 명확

---

## 📌 4. 성능 관련 주의할 점 (둘 모두 공통)

- **OFFSET이 클수록 느려진다**
    
    예: `LIMIT 1000000, 10` → 처음 100만 건을 스캔한 후 그다음 10건만 반환
    
    → MySQL은 OFFSET만큼 "건너뛰는" 최적화가 거의 없음
    
- 대용량 페이지네이션에서는 **커버링 인덱스 + SEEK 방식**을 쓰거나
    
    `WHERE id > ? LIMIT n` 같은 방식으로 최적화하는 게 좋다.
    

---

## ✅ 정리

- **`LIMIT offset, row_count` 와 `LIMIT row_count OFFSET offset` 은 성능·동작 동일**
- 차이는 오직 **문법과 가독성, 이식성**
- OFFSET 값이 커지면 둘 다 성능 저하 있음 (이건 공통 문제)
