## 브라우저가 `index.html`을 해석할 때 **HTML → CSS → JS** 동작 순서

### 1. HTML + CSS + JS 기본 로딩 과정
예를 들어 `index.html`이 이렇게 생겼다고 가정해보자

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <title>Document</title>
  <link rel="stylesheet" href="style.css">
  <script src="app.js" defer></script>
</head>
<body>
  <h1>Hello</h1>
</body>
</html>
```

#### (A) 브라우저 동작 순서
1. **HTML 파싱 시작**
    - 브라우저는 `index.html` 문서를 위에서 부터 한 줄씩 해석(파싱)
    - DOM(Document Object Model)을 만듦

2. **`<link rel="stylesheet">` 발견 → CSS 로딩**
    - 브라우저는 `style.css`를 **다운로드 시작**
    - CSS는 DOM과 결합해서 렌더 트리(Render Tree)를 생성
    - CSS 다운로드가 끝날 때까지 DOM이 멈추지는 않음(HTML 파싱은 계속)
    - 하지만 **렌더링(화면 표시)은 CSS 파싱이 끝나야 시작됨.*

3. **`<script>` 발견 → JS 로딩**
    - `defer`나 `async`가 없으면 → HTML 파싱이 멈추고 JS 다운로드 & 실행 먼저
    - `defer` → HTML 파싱은 계속 진행, JS는 다운로드만 하고 **DOM 파싱 완료 후 실행**
    - `async` → HTML 파싱은 계속 진행, JS는 다운로드 끝나는 즉시 실행 (순서 X)

4. **HTML 파싱 완료 → DOM 완성**
    - CSSOM(CSS Object Model)도 준비됨.
    - 이 시점에 **렌더 트리(Render Tree)** 생성 가능.

5. **렌더 트리 기반으로 레이아웃 계산 & 화면 그리기(Paint)**
    - DOM + CSSOM → Render Tree
    - Render Tree → Layout → Paint

6. **JavaScript 실행 (defer/DOMContentLoaded 이벤트)**
    - `defer` 스크립트가 실행됨.
    - 이 시점에 DOM 조작 가능.

#### (B) 순서 요약
- **HTML 파싱 → CSS 다운로드 & 파싱 → JS 실행 시점(속성 따라 다름) → 렌더링(Render Tree → Paint)**

|리소스|다운로드|실행 시점|
|---|---|---|
|HTML|위에서 아래로 파싱|즉시 (DOM 생성)|
|CSS|HTML 파싱 중 병렬 다운로드|파싱 완료 후 렌더링에 적용|
|JS (일반)|HTML 파싱 중 다운로드|다운로드 즉시 실행 → 파싱 중단|
|JS (defer)|HTML 파싱 중 다운로드|DOM 파싱 완료 후 실행|
|JS (async)|HTML 파싱 중 다운로드|다운로드 끝나는 즉시 실행 (순서 보장X)|

### 2. 핵심 이해 포인트
- **CSS**: DOM 파싱과 병렬로 다운로드, 화면 표시 전에 반드시 완료되어야 함.
- **JS**: DOM 파싱을 막을 수도 있고(`일반 script`), 지연 실행(`defer`), 비동기 실행(`async`)도 가능.
- **최종 렌더링**: DOM + CSSOM 둘 다 완성된 후 Render Tree가 만들어지고 화면에 그림(Paint).

👉 정리하면,
- **HTML이 먼저 → CSS는 병렬 로드 후 렌더링에 참여 → JS는 속성(defer/async)에 따라 실행 시점이 달라짐**

## `defer`와 `async` 차이
`<script>` 태그에서 `defer`와 `async`는 **외부 JavaScript 파일을 불러올 때 실행 시점**을 제어하는 속성

### 1. 기본 동작 (`<script src="...">`만 썼을 때)

- HTML은 위에서 아래로 순차적으로 해석(rendering)되다가 `<script>`를 만나면 **해당 JS 파일을 다운받고 실행할 때까지 렌더링이 멈춤**.
    → 화면 표시가 지연될 수 있음.
    
---
### 2. `defer`

```html
<script src="js/basic.js" defer></script>
```
- **HTML 파싱과 JS 다운로드를 동시에 진행**
- 하지만 **실행은 HTML 파싱이 끝난 뒤(DOM 생성 완료 직후)** 실행됨
- 여러 개의 `defer` 스크립트가 있으면 **작성된 순서대로 실행됨**

👉 특징:
- DOM이 완성된 상태에서 실행되므로 `document.getElementById` 같은 DOM 조작 코드에 안전함
- 보통 `<head>` 안에 넣는 경우가 많음

---
### 3. `async`

```html
<script src="js/basic.js" async></script>
```
- **HTML 파싱과 JS 다운로드를 동시에 진행**.
- JS 파일이 **다운로드가 끝나는 즉시 실행**됨(HTML 파싱 도중이라도 실행)
- 실행 순서는 보장되지 않음 → 다른 `async` 스크립트보다 먼저/나중 실행될 수 있음

👉 특징:
- 주로 **광고, 분석 스크립트(Google Analytics 등)** 같이 독립적으로 동작하는 JS에 사용
- DOM 의존성이 큰 코드에는 적합하지 않음

### 4. 비교 정리

|속성|다운로드 시점|실행 시점|실행 순서|활용 예시|
|---|---|---|---|---|
|없음|HTML 파싱 중단|즉시 실행|작성 순서|옛날 방식(지연 발생)|
|**defer**|HTML 파싱과 병렬|파싱 완료 후|작성 순서 보장|DOM 조작 스크립트|
|**async**|HTML 파싱과 병렬|다운로드 끝나면 즉시|순서 보장 X|광고, 통계, 독립 모듈|

---
👉 한 줄 요약:
- ==**defer** = DOM 파싱 끝난 뒤 실행, 순서 보장==
- ==**async** = 다운로드 끝나자마자 실행, 순서 보장 안 됨==

# 호이스팅 (Hoisting) / 렉시컬 스코프 (Lexical Scope)

## 1. 호이스팅 (Hoisting)

### ✅ 정의
- **호이스팅(Hoisting)**: 변수 선언과 함수 선언이 **실행 전에 메모리에 먼저 올라가는 현상**.
- 즉, 코드가 실행되기 전에 **변수 선언/함수 선언이 스코프의 최상단으로 끌어올려진 것처럼 동작**한다.

---
### ✅ 변수 호이스팅

```jsx
console.log(a); // undefined
var a = 10;
console.log(a); // 10
```

➡ 실행 과정 (자바스크립트 엔진 내부 동작):
```jsx
var a;            // 선언만 먼저 끌어올림
console.log(a);   // undefined
a = 10;           // 초기화
console.log(a);   // 10
```

⚠️ 주의:
- `var`는 **선언만 호이스팅**되고, 값 할당은 호이스팅되지 않음
- `let`, `const`는 호이스팅되지만 ==**TDZ(Temporal Dead Zone)==** 때문에 선언 전 접근하면 에러 발생
```jsx
console.log(b); // ReferenceError
let b = 20;
```

### ✅ 함수 호이스팅

```jsx
sayHello(); // "Hello"

function sayHello() {
  console.log("Hello");
}
```

➡ 함수 선언문(`function`)은 전체가 호이스팅됨 → 선언 전에도 호출 가능.

⚠️ 함수 표현식은 다름:

```jsx
sayHi(); // TypeError: sayHi is not a function

const sayHi = function() {
  console.log("Hi");
}
```

## 2. 렉시컬 스코프 (Lexical Scope)

### ✅ 정의

- **렉시컬 스코프(Lexical Scope)**: 함수가 **선언된 위치**에 따라 스코프(변수의 유효 범위)가 결정되는 것.
- 실행하는 위치가 아니라 **코드를 작성한 위치**가 스코프를 결정한다.

---
### ✅ 예시

```jsx
var x = 1;

function outer() {
  var x = 10;
  function inner() {
    console.log(x);
  }
  inner();
}
outer(); // 10
```

➡ `inner()` 함수는 **선언된 위치(outer 안)**를 기준으로 상위 스코프(outer → 전역)를 찾음.

➡ 실행 위치가 아니라 **선언 위치**가 중요하다.

---
### ✅ 실행 위치에 따른 오해 방지

```jsx
var y = 100;

function foo() {
  console.log(y);
}

function bar() {
  var y = 200;
  foo();
}

bar(); // 100 (200 아님!)
```

➡ `foo`는 전역에서 선언됨 → 전역 스코프의 `y=100`을 참조.

➡ **호출된 위치(bar 안)**이 아니라 **선언된 위치(전역)** 기준으로 스코프를 결정.

---
## 3. 호이스팅 & 렉시컬 스코프 함께 이해하기

```jsx
function outer() {
  console.log(a); // undefined (호이스팅으로 선언만 됨)
  var a = 5;

  function inner() {
    console.log(a); // 5 (렉시컬 스코프: outer의 a 참조)
  }
  inner();
}
outer();
```

## 4. 정리 표

|개념|의미|특징|주의할 점|
|---|---|---|---|
|**호이스팅**|선언이 실행 전에 메모리에 올라가는 현상|`var`, `function` 선언문 적용|`let`, `const`는 TDZ 때문에 선언 전 접근 불가|
|**렉시컬 스코프**|함수의 스코프는 선언된 위치로 결정됨|실행 위치와 무관|클로저(closure)의 기본 원리|
