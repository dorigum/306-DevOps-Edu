# React

#### 장점
- 빠른 업데이트와 렌더링 속도 -> 가상 DOM(Virtual DOM)

- 컴포넌트 기반의 구조
	- 선언형 프로그래밍 – 목적만 깔끔하게 명시하여 코드가 간결함
	- React에서는 모든 페이지가 컴포넌트(구성요소)로 구성된다.
	- 하나의 컴포넌트는 또 다른 여러 개의 컴포넌트 조합으로 구성될 수 있다.
	- 마치 레고 블록들이 모여서 하나의 완성된 모형이 되는 것과 비슷하다.

- 재사용성
	- 다른 모듈에 의존성을 낮추고 호환성 문제가 발생하지 않도록 개발하여 유지보수가 용이하게 만드는 것
	- React는 컴포넌트를 개발할 때 항상 쉽고 재사용이 가능한 형태로 개발한다.

- 든든한 지원군
	- 전세계 최대 IT기업 중 하나인 메타(구 페이스북)에서 시작한 프로젝트로 꾸준히 버전 업데이트가 이루어지며 발전

- 활발한 지식 공유와 커뮤니티
- 모바일 앱 개발 가능
	- 리액트 네이티브라는 모바일 환경 UI프레임워크를 사용하여 모바일 앱 개발 가능

## React Component
- 어떠한 속성들(props)을 입력으로 받아서 그에 맞는 React Element를 생성하여 리턴해준다.
- 컴포넌트는 리액트의 핵심 개념 중 하나이다.
- HTML에서 화면을 구성하는 요소(버튼, 링크, 이미지 등)로 볼 수 있다.
- 프로젝트 규모가 커져도 이미 작성한 컴포넌트를 재사용한다면 개발 속도를 빠르게 할 수 있으며, 유지 보수에도 유리하다. 
- 컴포넌트는 javascript 파일 내의 하나의 함수로 작성하거나 javascript 파일 자체로도 작성할 수 있다.

#### - 클래스 컴포넌트
- **클래스 컴포넌트**는 ES6 클래스를 활용하여 정의
- `React.Component` 를 상속받아 작성(Java 오버라이딩과 비슷)
- 컴포넌트의 상태(state)를 관리할 수 있다.
- 생명주기 메서드를 사용할 수 있다.

#### - 함수형 컴포넌트
- **함수로 정의**
- 상태와 생명주기 메서드를 사용할 수 없었지만, **React Hooks 도입**으로 가능해졌다.
- 간결하고 가독성이 좋다.
- 클래스 컴포넌트보다 성능이 약간 우수하다.

|               | 클래스 컴포넌트         | 함수형 컴포넌트                         |
| ------------- | ---------------- | -------------------------------- |
| **정의 방식(문법)** | ES6 클래스로 정의      | 함수로 정의                           |
| **상태 관리**     | 가능(this.state)   | React Hooks 사용 가능(useState)      |
| **생명 주기 메소드** | 사용 가능            | React Hooks 사용 가능(useEffect)     |
| **가독성**       | 일반적이고 명확한 구조     | 간결하고 구독성이 좋음                     |
| **성능**        | 일반적인 성능          | 성능이 약간 우수                        |
| **사용 시기**     | 초기의 React        | 상태가 필요하지 않는 경우, 간단한 UI 표현        |
| **장점**        | 복잡한 로직 처리에 적합    | 간결하고 가독성이 좋음                     |
| **발전 과정**     | 초기 React 컴포넌트 방식 | React 16.8에서 Hooks 도입으로 활용 범위 확장 |

#### 작성 방법
- 컴포넌트를 어느정도 수준까지 추출하는 것이 좋은 지에 대한 정해진 기준은 없지만,
	- **==기능단위로 구분하는 것이 좋고
	- ==**나중에 곧바로 재사용이 가능한 형태로 추출하는 것이 좋다.==**
- 함수 형식으로 컴포넌트를 작성하며, 함수 이름은 파일 명과 동일하게 하며, 함수가 두 개 이상인 경우에는 다른 이름으로 작성

#### JSX(JavaScript XML) 파일
- JSX는 리액트에서 사용하는 파일로 JavaScript 코드 안에 HTML과 유사한 코드를 작성할 수 있게 해준다.
- 리액트 컴포넌트를 정의하고 UI를 구성하는데 매우 유용하다.
- 브라우저에서 직접 실행되지 않으며, Babel과 같은 변환기를 통해 Javascript로 변환 후 실행된다.
- 리액트 자체 빌드도구 덕분에 `*.js` , `*.jsx`로 작성해도 아무런 문제없이 실행은 가능하나 jsx파일로 컴포넌트 만들 것을 권장한다.

##### ✅예시_컴포넌트 생성(`Component1.jsx`)
```jsx
// 함수 정의
function Component1() {
	// 출력할 내용을 return() 에 작성
	return(
		// 출력할 내용을 감싸기 위한 태그
		<>
			<h2>Hello Component1</h2>
		</>
	)
}

// 내보내기를 위한 문법
export default Component1;
```

##### ✅컴포넌트 적용
```jsx
// App.jsx로 돌아와서 Component1.jsx import
import Component1 from './Component1'
// return() 내부의 <></> 태그 내에 Component1 작성
```

##### ✅컴포넌트에 함수가 2개 이상인 경우
- `Component2.jsx` 를 만들고 Fun1, Fun2 함수 작성
- 함수를 각각 export 하기 위해 함수 이름 앞에 `export` 작성
```jsx
export function Func1() {
	return (
		<>
			<h2>hello Func1</h2>
		</>
	);
}

export function Func2() {
	return (
		<>
			<h2>hello Func2</h2>
		</>
	);
}
```

##### ✅하나의 함수에 `default` 를 적용하는 경우
```jsx
export default function Func1() {
	return (
		<>
			<h2>hello Func1</h2>
		</>
	);
}

export function Func2() {
	return (
		<>
			<h2>hello Func2</h2>
		</>
	);
}
```

##### ✅`default` 를 작성한 함수는 `{}` 로 import 하지 않으며, 아래와 같이 `import` 문장 작성
```jsx
import Func1 from "./Component2";
import {Func2} from "./Component2";
```

# 권장되는 함수 선언 스타일
> 언제, 어떤 맥락에서 함수를 선언하느냐에 따라 권장되는 스타일이 달라진다.
**순수 자바스크립트**와 **React** 각각의 관점에서 정리해보자.

## 1. 순수 자바스크립트에서의 함수 선언 방식

### (1) 선언적 함수 (Function Declaration)

```jsx
function add(a, b) {
  return a + b;
}
```

- **호이스팅(hoisting)**: 함수 전체가 끌어올려지므로, 선언 이전에도 호출 가능.
- **장점**: 전역적 유틸리티나 라이브러리 함수처럼 “어디서든 불러 쓸” 함수에 적합.
- **단점**: 호이스팅으로 인한 예측 불가능한 동작 위험
	- (특히 같은 이름의 함수가 나중에 덮어써지는 경우)

**요약**: 작은 스크립트나 전통적인 JS 코드에서는 여전히 자주 사용되지만, ES6 이후엔 함수 표현식이 선호됨

---
### (2) 함수 표현식 (Function Expression)

```jsx
const add = function(a, b) {
  return a + b;
};
```

혹은 **화살표 함수(Arrow Function)** 로:

```jsx
const add = (a, b) => a + b;
```

- **호이스팅 없음**: 선언 이전에 호출 불가 → 예측 가능한 실행 순서
- **스코프 안전성**: 블록 범위(`let`, `const`) 안에서만 유효.
- **this 바인딩 주의**: 화살표 함수는 자신만의 `this`를 가지지 않음
	- 클래스 메서드나 콜백에서 특히 유용

**요약**: ES6+ 환경과 모듈 시스템에서는 대부분 **함수 표현식**(특히 화살표 함수)을 사용
👉 즉, “현대적 자바스크립트 스타일”에서는 **표현식 함수가 기본값**


## 2. React에서의 함수 선언 스타일

### (1) 함수형 컴포넌트 (Function Component)

React의 컴포넌트는 사실상 “함수”이지만, 여기서는 **선언적 함수 스타일**이 더 흔하다.

```jsx
function Header() {
  return <h1>Hello</h1>;
}
```

### 이유:

1. **React DevTools 디버깅 시 이름이 자동 인식**됨
    → `const Header = () => {}`로 하면 변수명 기반 인식이지만, 선언형은 더 명확.

2. **React 공식 문서**도 대부분 이 스타일 사용.

3. **React.memo, forwardRef 등 HOC(Higher-Order Component, 고차 컴포넌트)**에서 읽기 쉽고 명시적임. (HOC는 **컴포넌트 로직을 재사용하기 위한 React의 고급 기술**)

4. `function` 키워드는 컴포넌트가 단순한 함수라는 것을 직관적으로 표현.


---
### (2) 화살표 함수 컴포넌트

```jsx
const Header = () => <h1>Hello</h1>;
```

### 사용 이유:
- 인라인 콜백이나 prop으로 넘기는 함수 정의 시 간결함.
- 함수형 프로그래밍 스타일을 유지하고 싶을 때.
- `useCallback` 등과 함께 쓸 때 가독성이 좋음.

### 주의:
- 클래스 바인딩 문제는 없지만, 렌더 시마다 재생성되는 함수를 prop으로 넘기면 **불필요한 리렌더**를 초래할 수 있음.

## 정리 비교표

|구분|형태|예시|특징 / 사용처|권장도|
|---|---|---|---|---|
|**JS 선언적 함수**|`function foo(){}`|전통적 함수|호이스팅 있음, 유틸리티 함수|⚪ 일반적|
|**JS 함수 표현식**|`const foo = function(){}`|모듈 내부 함수|예측 가능, 스코프 안전|✅ 권장|
|**JS 화살표 함수**|`const foo = () => {}`|콜백, 단일행 함수|짧고 `this` 없음|✅ 권장|
|**React 선언형 컴포넌트**|`function Header(){}`|컴포넌트 정의|React 공식 스타일|✅ 권장|
|**React 화살표 컴포넌트**|`const Header = () => {}`|함수형 스타일|간결하지만 재생성 주의|⚪ 선택적|

---
## 결론

- **순수 자바스크립트**:
    ➤ **`const` 함수 표현식 (특히 화살표 함수)** 가 주류이자 권장 방식

- **React**:
    ➤ **컴포넌트 정의는 `function ComponentName()` 형태**가 공식 문서 기준 권장
    ➤ **콜백이나 내부 핸들러는 화살표 함수**로 작성하는 것이 일반적

## 예제 1: React 컴포넌트에서의 일반적인 함수 사용 패턴

```jsx
//  React 권장 스타일
function Counter() {
  // 🔹 상태(state)
  const [count, setCount] = React.useState(0);

  // 🔹 이벤트 핸들러 — 화살표 함수로 작성 (권장)
  const handleIncrease = () => setCount(count + 1);
  const handleDecrease = () => setCount(count - 1);

  // 🔹 내부 유틸리티 함수 — 화살표 or 일반함수 모두 가능
  function formatCount(value) {
    return value < 10 ? `0${value}` : value;
  }

  // 🔹 렌더링
  return (
    <div style={{ textAlign: "center" }}>
      <h2>카운터</h2>
      <p>{formatCount(count)}</p>
      <button onClick={handleDecrease}>-</button>
      <button onClick={handleIncrease}>+</button>
    </div>
  );
}
```

### 해설

|구분|형태|이유|
|---|---|---|
|`function Counter()`|선언적 함수|React 컴포넌트 정의 시 권장 방식|
|`const handleIncrease = () => {}`|화살표 함수|이벤트 핸들러는 재사용 필요 없고 간결함|
|`function formatCount()`|선언적 함수|내부에서 호출 순서 예측 가능, 디버깅 명확|

## 예제 2: 컴포넌트 안에서 함수가 “props로 전달”될 때

```jsx
function Parent() {
  const [text, setText] = React.useState("Hello");

  // 🔹 콜백을 자식에게 넘길 때는 useCallback + 화살표 함수로
  const handleChange = React.useCallback((newValue) => {
    setText(newValue);
  }, []);

  return (
    <div>
      <h3>부모 컴포넌트</h3>
      <Child text={text} onTextChange={handleChange} />
    </div>
  );
}

function Child({ text, onTextChange }) {
  return (
    <div>
      <inputvalue={text}
        onChange={(e) => onTextChange(e.target.value)}
      />
    </div>
  );
}
```

### 포인트
- **props로 전달되는 함수는 `useCallback` + 화살표 함수** 조합이 매우 일반적이다.
    → 이유: 렌더링 때마다 새로 만들어지는 함수 참조를 방지해 불필요한 리렌더를 줄이기 위해서.

- React에서는 이 패턴이 **성능 최적화의 기본기이다.**

## 예제 3: 여러 함수 스타일 혼용 예시

```jsx
function TodoList() {
  const [todos, setTodos] = React.useState(["공부하기", "운동하기"]);

  //  화살표 함수 (이벤트 핸들러)
  const addTodo = () => setTodos([...todos, "새 할 일"]);

  //  일반 함수 (비즈니스 로직, 재사용성)
  function removeTodo(index) {
    setTodos(todos.filter((_, i) => i !== index));
  }

  return (
    <div>
      <h2>오늘의 할 일</h2>
      <button onClick={addTodo}>추가</button>
      <ul>
        {todos.map((todo, index) => (
          <li key={index}>
            {todo}{" "}
            <button onClick={() => removeTodo(index)}>삭제</button>
          </li>
        ))}
      </ul>
    </div>
  );
}
```

## 정리

| 상황                | 추천 함수 스타일                                 | 이유                       |
| ----------------- | ----------------------------------------- | ------------------------ |
| **컴포넌트 선언**       | `function ComponentName()`                | 공식 문서 권장, DevTools 인식 용이 |
| **이벤트 핸들러**       | `const handleX = () => {}`                | 짧고 명료, this 바인딩 문제 없음    |
| **콜백(props로 전달)** | `const handleX = useCallback(()=>{}, [])` | 리렌더 방지, 최적화              |
| **내부 유틸리티/헬퍼 함수** | `function helper()`                       | 호출 순서 명확, 디버깅 편리         |

# “이미지 최적화”란?

## 1. “이미지 최적화”란 무엇인가?

>React 빌드 과정에서 Webpack/Vite는 이미지를 단순히 복사하는 것이 아니라 **다음과 같은 최적화 작업**을 자동으로 수행한다.

### ✔ 1) 이미지 압축(Compression)
- JPG, PNG 등 이미지 용량을 가능한 한 줄임
- 브라우저 요청 속도 증가 → 웹사이트가 더 빨리 로딩됨

### ✔ 2) 필요 시 WebP 등 더 효율적인 형식으로 변환
- 현대적 포맷(WebP, AVIF)이 더 용량이 작음
    _(프로젝트 설정에 따라 자동 변환 가능)_

### ✔ 3) 작은 이미지 → Base64로 변환(인라인 처리)
- 매우 작은 아이콘은 `.svg`, 작은 `.png`를 base64 문자열로 JS/CSS 내부에 넣어버림
- HTTP 요청 수가 줄어들어 성능 향상

### ✔ 4) 사용되지 않는 이미지 제거(Tree-shaking)
- 코드에서 **import되지 않는 이미지**는 빌드 결과물에 포함되지 않음
- 불필요한 용량 절약

즉, **이미지를 자동으로 가장 빠르게 로딩될 수 있게 가공하는 과정 전체를 최적화라고 부른다.**


---
## 2. 번들링된 후 이미지 이름이 랜덤하게 바뀌는 이유 (해시값)

예시:

```
logo.png → logo.4fa3bc12.png
```

이런 랜덤 문자열(정확히는 **content hash**)은 **캐싱 전략**을 위한 것이다.


---
## 3.왜 이름을 랜덤하게(해시값 변화) 만들까?

브라우저는 성능 향상을 위해 **정적 파일을 캐싱한다.**
즉, 한 번 받은 `logo.png`를 다음에도 계속 그대로 사용함.

하지만 문제가 하나 있다:

> ✔ 이미지 내용을 수정했는데, 브라우저는 기존 캐시된 파일을 계속 사용해서 변경이 반영되지 않는 문제 발생한다.

👉 이를 막기 위해 Webpack/Vite는 파일의 **내용(content)** 기반으로 해시를 붙인다.


---
## ✔ 해시 파일명 방식의 장점

### 1) **내용이 변경되면 해시가 달라짐 → 자동으로 캐시 무효화**
- 새로운 파일 이름이 만들어짐

- 브라우저가 새로운 파일을 다시 받음
    → **사용자가 항상 최신 파일을 보게 됨**


### 2) **내용이 변경되지 않으면 해시도 유지 → 캐시 유지**
- 불필요한 네트워크 요청이 사라져 성능 상승

---
## 해시 파일명이 하는 역할

|기능|설명|
|---|---|
|**캐시 무효화(Cache Busting)**|이미지 수정 시 자동으로 새로운 파일 이름 생성|
|**로드 속도 개선**|이미지가 바뀌지 않았다면 캐시를 계속 사용|
|**빌드 안정성**|파일 이름 충돌 방지|
|**CDN 친화적**|CDN에 캐시를 매우 길게(1년) 유지해도 문제 없음|
- **최적화**: 이미지 압축, 포맷 변환, 작은 파일 인라인 처리, 미사용 파일 제거 등 성능 최대화
- **랜덤/해시 파일명**: 캐싱을 안정적으로 관리하기 위한 "캐시 버스팅(Cache Busting)" 기술

# ⭐React의 State란?⭐

- 컴포넌트 내부에서 관리되는 동적인 데이터이다.
- 컴포넌트의 상태를 나타내며, 사용자 입력 등에 따라 변경될 수 있다.
- ==state가 변경되면 리액트는 자동으로 해당 컴포넌트와 자식 컴포넌트를 다시 렌더링 한다.==
- useState() 훅을 통해 state 상태를 관리할 수 있다.

### 💡 리액트 훅(Hooks)
- 함수형 컴포넌트에서 상태와 생명주기 기능을 사용할 수 있도록 해주는 기능으로, 다양한 종류의 훅이 있다.
- 대표적으로 `useState`, `useEffect`, `useContext`, `useReducer`, `useCallback`, `useMemo`, `useRef` 등이 있다.

### State 선언
- state 변수와 state 값을 변경할 함수 이름을 함께 선언하며, 우변에는 초기 값을 useState() 의 괄호 안에 작성한다.
- **useState 사용을 위해 useState 모듈 추가**

```jsx
Import React, { useState } from'react';

/* [ 사용법 ] */
const [상태 값 저장 변수, 상태 값 갱신 함수] = useState(상태 초기 값);

/* [ 예제 ] */
const [no, setNo] = useState(0);
```
