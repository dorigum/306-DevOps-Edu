# JDBC
## JDBC(Java Database Connectivity) API
![[20260309_102945.jpg]]
### JDBC (Java Database Connectivity)란?
- 자바에서 데이터베이스에 접근하기 위한 표준 API
- 다양한 DBMS(MySQL, Oracle, PostgreSQL 등)와 독립적으로 동작할 수 있게 해줌
- JDBC는 드라이버 매니저(DriverManager)를 통해 데이터베이스 벤더가 제공하는 드라이버를 로드하고, SQL 실행을 가능하게 함.

### JDBC 아키텍처
1. JDBC API
   : DriverManager, Connection, Statement, ResultSet 등의 인터페이스 제공

2. JDBC Driver
   : DB 벤더(예: MySQL, Oracle)가 제공하는 실제 구현체

3. 드라이버 종류
   Type 1: JDBC-ODBC Bridge (거의 사용 X)
   Type 2: Native API Driver
   Type 3: Network Protocol Driver
   Type 4: Thin Driver (순수 자바 기반, 가장 많이 사용)

```JAVA
import java.sql.*;
```

# SQL Injection공격

**MySQ과 Oracle은 타입 변환 규칙 달라서 MySQL 은 예외가 나지 않는 것**

1. **PreparedStatement는 값과 SQL을 분리**
    
    `where sal > ?` 에서 `?`는 **SQL 구문이 아니라 값**으로 전달된다.
    
    즉 `"2000 or 1=1"`은 SQL로 해석되지 않고 **그냥 문자열 값**
    
2. **Oracle vs MySQL의 암묵적 타입 변환 차이**
    
    - Oracle: 숫자 비교(`sal > ?`)에 문자열이 들어오면 **문자열 전체를 NUMBER로 변환**하려고 시도합니다.
        
        `"2000 or 1=1"` → 숫자로 변환 불가 → `ORA-01722: invalid number` 예외.
        
    - MySQL: 숫자 컨텍스트에서 문자열을 보면 **앞부분의 숫자만 읽어 숫자로 변환**하고 뒤의 “쓰레기”는 무시하는, 매우 관대한 규칙을 쓴다
        
        `"2000 or 1=1"` → **2000으로 변환** → `sal > 2000`으로 평가 → 예외 없이 정상 실행.
        
        (이때 드라이버/모드에 따라 “Truncated incorrect DOUBLE value …” 같은 **경고**가 생길 수 있지만 보통 **에러는 아닙니다**.)
        
    
    즉, **인젝션은 실패**했지만(MySQL이 `or 1=1`을 코드로 보지 않음), **형 변환이 관대**해서 예외가 안 난 것이다.
    
3. 올바른 타입으로 바인딩 → 타입을 맞춰 바인딩
```java
int cutoff = Integer.parseInt(param.trim()); // 숫자 아님 -> NumberFormatException
ps.setInt(1, cutoff);
```
- DB마다 다른 암묵적 변환(Oracle의 예외, MySQL의 관대함)에 의존하지 않게 되고, **의도도 더 명확해진다.**
